# A.L.I.S.T.A.I.R.E. charter candidate

A.L.I.S.T.A.I.R.E. is a proposed female-presenting research-agent architecture composed of interoperable Quantum State Objects and bounded supporting services. The portfolio is in a **documentation, governance-consolidation, contract-sequencing, and repository-consolidation phase**. No executable AGI, consciousness, sentience, unrestricted autonomy, device-control service, payment authority, persistent self-improvement, or production deployment is established.

## Decision status

**Blocked:** two repositories currently overlap as the constitutional identity:

- `aevespers2/ALISTAIRE-` contains the established product directive, task chain, release gates, punch list, coordination history, and consolidated governance candidate.
- `aevespers2/Alistaire-agi` contains the proposed `alistaire-qso` package name, compatibility-landing candidate, migration-manifest example, and historical topic taxonomy.

The immediate work is not runtime implementation. It is acceptance or revision of five constitutional decisions:

1. canonical charter and repository identity;
2. neutral, non-operational contract steward;
3. canonical bytes and identity primitives;
4. independent authority and recovery roots;
5. portfolio incident, freeze, rollback, and claim-withdrawal command.

These decisions break circular dependencies without activating credentials, online services, enrolled devices, autonomous execution, or deployment.

After approval, a researcher or Architect should be able to answer:

- What is A.L.I.S.T.A.I.R.E. for, and what is explicitly outside scope?
- Which repository is the constitutional source, and how is the non-canonical history preserved?
- Which neutral authority owns common IDs, envelopes, canonical bytes, reason codes, compatibility, migration, and fixtures?
- How do Repositories `0` and `1` secure, recover, and maintain an owned device without self-authorization?
- Which repository owns identity, runtime, collaboration, source evidence, temporal interpretation, domain interpretation, transport, review, payments, conformance, and publication?
- Where do independent approval, consent, privacy, freeze, correction, revocation, recovery, and rollback apply?
- Which pairwise and triple-overlap witnesses are required before components can compose?
- What deterministic evidence is required before any capability, security, compatibility, or release claim?

## Portable security foundation

Repositories `0` and `1` are intended to be the first A.L.I.S.T.A.I.R.E. components installed on a new, recovered, replaced, reset, or suspect owned device.

- **Repository `0`** is the candidate bootstrap, host-observation orchestration, proposal, bounded-execution, verification, and maintenance plane.
- **Repository `1`** or a successor is the candidate independent quarantine, capability, revocation, canonical-state, checkpoint, and recovery authority.

The documented route is:

`approved observation → Repository 0 local proposal → Repository 1 quarantine → independent decision → narrow expiring capability → bounded execution → receipt → Repository 1 reconciliation`

Local staging, command success, a signed local artifact, or an interface interaction is not automatic canonical acceptance. Unsupported or inaccessible state remains `UNKNOWN`.

## Proposed product mode

The safest initial mode remains a **local, simulation-only research orchestrator**. It loads fixed contract and identity fixtures, evaluates bounded scenarios, records provenance and uncertainty, and produces reviewable proposals. It has no network, real credentials, external device access, external writes, payments, or self-authorized execution.

A future interactive companion, portable device agent, or broader research assistant requires separate approval for identity expectations, privacy, consent, retention, authority, evaluation, deployment, incident response, and recovery.

It has no independent credentials, device-control, merge, release, deployment, payment, canonical-state, or self-expansion authority.

Consequential authority remains explicitly granted, narrowly scoped, revocable, auditable, recoverable, and subject to required independent human approval.

## Evidence vocabulary

Every statement should be classified as:

- **Observed:** directly present in a repository, commit, workflow, artifact, or accepted contract.
- **Implemented:** code or data exists, but may still require independent validation.
- **Verified:** reproduced at an immutable commit with retained evidence.
- **Proposed:** design requiring approval or implementation.
- **Hypothesis:** research idea without accepted evidence.
- **Prohibited:** outside the current authority boundary.
- **Corrected:** a prior statement or record has an accepted correction lineage.
- **Revoked:** prior authority or trust has been withdrawn.
- **Withdrawn:** a public claim or candidate is no longer supported.

Repository names, documentation, empty pages, ceremonial titles, successful commands, signatures, interfaces, dependencies, and architecture diagrams are not implementation or authorization evidence.
