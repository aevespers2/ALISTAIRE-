# Alistaire charter candidate

Alistaire is a proposed female-presenting research agent architecture composed of interoperable Quantum State Objects. The project is currently in a **documentation, governance-consolidation, and repository-consolidation phase**. No executable AGI, consciousness, sentience, unrestricted autonomy, persistent self-improvement, or production deployment is established.

## Decision status

**Blocked:** two repositories currently overlap:

- `aevespers2/ALISTAIRE-` contains the established product directive, task chain, release gates, punch list, coordination history, and consolidated governance candidate.
- `aevespers2/Alistaire-agi` contains the proposed `alistaire-qso` package name and a broad, mostly empty documentation scaffold.

The documentation here prepares the evidence needed to select one canonical repository and adopt one governance model without silently discarding either history. It does not make the protected-branch approval decision.

## Intended outcome

After approval, a researcher or Architect should be able to answer:

- What is Alistaire for, and what is explicitly outside scope?
- Which QSO subsystem owns identity, memory, goals, policy, planning, evidence, perception, and tool proposals?
- Which repository owns each contract and implementation surface?
- Who owns autonomous planning, canonical state, capability issuance, verification, merge, release, deployment, incident response, emergency stop, and rollback?
- What information may be learned, persisted, shared, or forgotten?
- Where do human confirmation, consent, freeze, shutdown, and rollback apply?
- What deterministic evidence is required before any capability claim or release?

## Proposed product mode

The safest initial mode is a **local, simulation-only research orchestrator**. It composes accepted QSO contracts, evaluates bounded scenarios, records provenance and uncertainty, and produces reviewable proposals. It has no network, credentials, external writes, or self-authorized execution.

A future interactive companion or broader research assistant would require a separate approval cycle covering identity expectations, privacy, consent, retention, tool authority, evaluation, deployment, and incident response.

## Governance posture

A.L.I.S.T.A.I.R.E. is the unifying portfolio objective; repositories are bounded subsystems. The governance candidate names **Cali Sanders Parker**, ceremonially titled **Calisandra, Queen of the Nymphs**, as the documentation-coherence and security-stewardship QSO. That role prepares governance proposals, detects authority conflicts, aligns documentation, and escalates security gaps. It has no independent credentials, merge, release, deployment, payment, canonical-state, or self-expansion authority.

The working model identifies Repository `0` as the autonomous-development orchestration candidate and Repository `1` or an approved successor as the independent canonical-state and capability-authority candidate. Consequential authority remains explicitly granted, revocable, auditable, and subject to required human approval.

## Evidence vocabulary

Every statement should be classified as one of:

- **Observed:** directly present in a repository, commit, workflow, artifact, or accepted contract.
- **Implemented:** code or data exists, but may still require independent validation.
- **Verified:** reproduced at an immutable commit with retained evidence.
- **Proposed:** design requiring approval or implementation.
- **Hypothesis:** research idea without accepted evidence.
- **Prohibited:** outside the current authority boundary.

Repository names, empty pages, ceremonial titles, future-facing terminology, and architectural diagrams are not implementation or authorization evidence.

## Release posture

The first possible release is `0.0.1-charter`. It is documentation-only and remains blocked until canonical repository selection, migration/provenance planning, governance and charter approval, capability-authority disposition, security/privacy review, updated exact-head documentation validation, artifact hashing, rollback evidence, and explicit approval are complete.
