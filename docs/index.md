# A.L.I.S.T.A.I.R.E. charter candidate

A.L.I.S.T.A.I.R.E. is a proposed research-agent architecture composed of interoperable Quantum State Objects and bounded supporting services. The portfolio is in a **documentation, governance-consolidation, contract-sequencing, canonicalization, semantic-partition, and repository-consolidation phase**. No executable AGI, consciousness, sentience, unrestricted autonomy, device-control service, payment authority, persistent self-improvement, or production deployment is established.

## Name

**A.L.I.S.T.A.I.R.E.** expands to **Adaptive Learning & Intelligence System for Trustworthy Autonomous Inference, Reasoning, and Evolution**.

The [name and identity guide](name-and-identity.md) defines every term, separates the display name from repository and package identities, provides public writing rules, and explains why “autonomous” means bounded internal inference rather than standing operational authority. Its status is `NAME_EXPANSION_DOCUMENTED_CANONICAL_REPOSITORY_UNSELECTED`.

## Capability roadmap

The [capability roadmap](capability-roadmap.md) turns forty desired features into a bounded architecture and review sequence. It organizes six capability families, maps candidate repository homes and unresolved gates, sequences stages R0–R5, records material composition obstructions, and requires correction and rollback rather than silent promotion.

Its status is `DOCUMENTED_CAPABILITY_ROADMAP_UNACCEPTED`. The competing flat feature source was reconciled under `RECONCILED_INTO_EXISTING_ROADMAP_PRESERVE_SOURCE_BRANCH`; it remains historical evidence rather than a parallel authority registry.

Documentation, repository assignment, interfaces, fixtures, branches, workflows, and passing tests do not establish implementation, ownership, compatibility, trust, acceptance, release, Pages publication, deployment, credentials, or operational authority.

## Runtime/Fabric semantic partition

The portfolio graph records `BLOCKED_ROLE_COLLISION`. `QuantumStateObjects` needs runtime-local event and execution records; `QSO-FABRIC` needs projection, collaboration, experiment, and aggregate-evidence records. Reusing labels across these levels can hide source sets, collapse record identity, inflate duplicate evidence, break correction propagation, and promote local success into aggregate or portfolio success.

The [runtime/Fabric namespace-partition packet](runtime-fabric-namespace-partition.md) distinguishes semantic classes and compares unselected profiles. It defines producer and source-set bindings, projection and aggregation receipts, ordering and replay rules, pairwise and triple-overlap witnesses, mixed-generation migration, correction, revocation, rollback, and failed-rollback handling.

The documented route remains:

`qsio-kernel → QuantumStateObjects → QSO-FABRIC → Repository 1`

Its safe disposition is `UNSUPPORTED_KERNEL_RUNTIME_ROUTE`. Direct identity aliasing remains rejected as `REJECT_DIRECT_IDENTITY_ALIAS`.

- Which neutral authority owns common IDs, envelopes, canonical bytes, reason codes, compatibility, migration, and fixtures?
- How are runtime-local records distinguished from Fabric projections, aggregates, and Repository `1` dispositions?
- Which repository owns identity, runtime, collaboration, source evidence, temporal interpretation, domain interpretation, transport, review, payments, conformance, and publication?
- Where do independent approval, consent, privacy, freeze, correction, revocation, recovery, and rollback apply?
- Which pairwise and triple-overlap witnesses are required before components compose?
- What evidence is required before any capability, security, compatibility, or release claim?

## Portable security foundation

Repositories `0` and `1` are intended to be the first A.L.I.S.T.A.I.R.E. components installed on a new, recovered, replaced, reset, or suspect owned device.

- **Repository `0`** is the candidate bootstrap, host-observation orchestration, proposal, bounded-execution, verification, and maintenance plane.
- **Repository `1`** or successor is the candidate independent quarantine, capability, revocation, canonical-state, checkpoint, and recovery authority.

The documented route is:

`approved observation → Repository 0 local proposal → Repository 1 quarantine → independent decision → narrow expiring capability → bounded execution → receipt → Repository 1 reconciliation`

Local staging, command success, a signed local artifact, or an interface interaction is not automatic canonical acceptance. Unsupported or inaccessible state remains `UNKNOWN`.

## Proposed product mode

The safest initial mode remains a **local, simulation-only research orchestrator**. It loads fixed contract and identity fixtures, evaluates bounded scenarios, records provenance and uncertainty, and produces reviewable proposals. It has no network, real credentials, external device access, external writes, payments, or self-authorized execution.

## Governance posture

A.L.I.S.T.A.I.R.E. is the unifying portfolio objective; repositories are bounded subsystems. The governance candidate assigns documentation coherence, architectural reconciliation, governance proposals, security-boundary review, and escalation duties without granting credentials, device control, merge, release, deployment, payment, canonical-state, or self-expansion authority.

Consequential authority remains explicitly granted, narrowly scoped, revocable, auditable, recoverable, and subject to required independent human approval.
